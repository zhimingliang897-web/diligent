# day 1 大模型底层原理与源码灵魂拷问 Q&A

**学习时间**：2026-02-20
**核心内容**：基于 MiniMind / LLaMA 架构的代码级硬核答疑，直击大模型面试高频考点。

---

## 1. 词表与上下文长度的本质是什么？

**Q：`vocab_size=6400` 是人定的还是机器算的？大模型到底是个什么任务？**
大模型在最底层，本质上是在做一道**拥有 6400 个选项的单项选择题**。

* 坑位数量（6400）是人类根据显卡和需求提前定死的。
* 里面装什么字/词根（Token），是统计算法（如 BPE）在跑语料时算出来的频率最高的片段。
  当大模型把数据跑到最后一步（LM Head）时，它的终极任务就是给这 6400 个选项打分（其实就是一个标准的超大规模**分类问题**，然后用交叉熵 Cross Entropy 算误差）。

**Q：`max_position_embeddings=32768`（最大上下文）是怎么确定的？**
这不是模型自己想的，而是**人类向现实（显存和算力）妥协设定的天花板**。
因为自注意力机制的计算复杂度是 $O(N^2)$，字数翻倍，计算量暴增四倍。32768 是在普通人能买得起的显卡（24G显存）和实际长文需求之间的一个极限折中点。

## 2. 谁决定了大模型什么时候“闭嘴”？

**Q：为什么有了 FFN 层，还要设定 `eos_token_id` (句子终止符)？句子长度是 FFN 决定的吗？**
**绝对不是！**
模型的任何层（包扣 FFN 和最后的 LM Head），每次无论算多久，输出的永远是一个固定维度（比如 6400 维）的打分表，它自己**不能**决定句子长短。
句子长短是由外面跑推理聊天的 Python 代码（也就是一个不断调用模型的 `while` 死循环，如 transformers 底层的 `generate` 函数）决定的。

触发生成停止只有三个原因（抢答机制）：

1. **自己说完了**：模型在打分表里算出来 `EOS`(比如 2 号) 得分最高，循环代码一看到 2 就直接 `break` 切断。
2. **强行拉闸**：达到了人设定的字数上限 `max_new_tokens`。
3. **脑袋撑爆了**：上下文超过了 `max_position_embeddings` 架构极限。

## 3. 面试必问：RMSNorm 为什么能取代 LayerNorm？为什么要留个可训练的 Gamma？

**Q：RMSNorm 比老前辈 LayerNorm 强在哪里？**
传统的 LayerNorm 归一化要做两件事：算均值去平移、算方差去缩放。
大佬们发现算均值不仅慢，而且在极其深的网络里对模型提升智商毫无帮助。于是 **RMSNorm 直接一刀砍掉了算均值和减均值的平移步骤，仅仅使用均方根（RMS）进行单纯的缩放。** 计算量大减，推理速度狂飙10%~30%，最后效果居然没下降。这就是“极致减肥去废操作”的典范。

**Q：既然用纯数学公式降维打击压扁了数据，为什么还要保留一个给模型训练的参数 $\gamma$（Gamma）？**
因为神经网络有一个核心矛盾：“既要整齐划一，又要保留个性”。
数学公式的归一化，是为了防止数据在几十层流转里爆炸或消失（统统压回 0~1 的死板模子里去）。
但如果压得太死，珍贵的高维个性特征就被淹没了。所以必须留下唯一的放缩参数 **$\gamma$**，让模型在训练时可以自我重塑姿态，决定哪些维度的特征需要再被放大提拉出来。如果不训练 $\gamma$ 强行全为 1，模型表达能力会断崖式下降。

## 4. GQA 是在解决什么绝症？

在大模型的自注意力层，最吃紧的是保存历史聊天记录的 **KV Cache 显存**。

* **MHA（全多头，历史老古董）**：8个Q头对应8个K头和8个V头。极度聪明，但显存瞬间爆炸。
* **MQA（多查询，极端省钱派）**：8个Q头共享可怜的1个K头和1个V头。速度极快显存极小，但是容易犯智障错误。
* **GQA（分组查询，主流王者）**：比如 LLaMA 和 MiniMind，8个Q分成两组，对应 2个K头和 2个V头。在不明显掉智商的前提下，大幅削减了 KV 内存。

**Q：代码里的 `repeat_kv` 是干什么的？**
在 GQA 里，因为 K 和 V 的头少（维度对不上 8组Q），底层无法做矩阵乘法。`repeat_kv` 就是极其粗暴的复印机，把 2个K 强行每人复制 4 份，假装成 8 份的厚度塞进去，跟 Q 的维度严丝合缝对齐进行开战。

## 5. SwiGLU 的维度风暴（FFN是怎么进行深思熟虑的？）

传统前馈网络是一条路走到黑的单调拉伸。**SwiGLU 则是引入了“门控（Gate）黑科技”的三明治双线网络**，负责让大模型产生极其复杂的“如果..那么..但是..”类的非线性逻辑分支。

**内部维度过山车（核心死记）：**
假设输入是 512 维：

1. **输入阶段**：兵分两路（明线跑 `up_proj`，暗线跑 `gate_proj`），双双从 512 维暴涨到比如 1408 维的宽广脑洞！
2. **绞肉机融合**：暗线过一下 `SiLU` 变种激活函数后，变成 1408 个数字开关，跟另外一路明线的 1408 个数字进行“哈达玛积 $\odot$ ”同位绞杀相乘。这就是非线性智能路障！
3. **输出收敛**：最后让 `down_proj` 把残余的 1408 维高阶混合物，再狠狠无情压缩映射回初始的 512 维度。**首尾宽度 512 必须严丝合缝，才能传给下一层。**

## 6. 一眼看穿训练底裤：模型是怎么做卷子的？（Shift 魔法）

**Q：为什么推理想输出短句子，而训练时 `input_ids` 和 `labels`（标签）的维度无论多少张永远是死死焊在一模一样的庞大固定维度？**

这里是新手最容易栽跟头的地方。推理是`while`循环蹦出的字，**训练其实是一把梭的“涂卡机批卷法”！**
为了压榨 GPU 极限性能，数据集都会被填出废话或者暴力切碎裁边，强行组成比如长 512 宽矩阵（`Batch Size * 512`）的双胞胎。$X$ 和 $\text{Label}$ 是完全等宽（通常内容也一模一样）的两块砖并排进入模型。

**神级操作：错位 `Shift` 法！**
进入 `forward` 打分大厅后，代码兵不血刃地做了一个极其狡猾的位移：

* 把模型拼死预测的打分表 `logits` 向左推一格。
* 把正确的标准答案 `labels` 向右推一格。

原本严丝合缝的两块 512 长张量，在交战时被强行错开了一个身位！
就在这一瞬间，它在底层悄无声息地并行化成了 511 道独立的“看当前全部上文，猜下一个字的完型填空抢答题对照组”！

**Q：不想让模型学提问部分的无用废墨水怎么办？**
工程师会在制作那两块砖的时候，把标签卷子里涉及提问、补齐空白（Pad）的格子全像涂改液一样盖上 `-100`。
在算错位对答案的交叉熵（`Cross_Entropy`）考官眼里，只要瞄到 `-100`，它直接变瞎跳过，这部分就算模型猜得像狗屎一样也永远不用扣罚智商成绩。这就是大模型的局部定向精微调（SFT）核心秘籍！

# Day 2 分词器 · 指令对齐 · 人类偏好优化的核心要点

**学习时间**：2026-02-21
**核心内容**：BPE 分词器的底层逻辑、SFT 指令微调的工程精髓，以及从 SFT 走向 RLHF 的全链路偏好对齐思路。

---

## 1. BPE 分词器：为什么不按字切、也不按词切？

**Q：直接按字母或单字切不行吗？**
两个极端都有致命缺陷：

- **按字母切**：序列极长，$O(L^2)$ 的注意力复杂度会直接炸显存；
- **按完整词切**：新词/罕见词（”绝绝子”、”VibeCoder”）词表必须巨大，而且必然有”未知词（UNK）”死局。

**BPE（字节对编码，1994 年算法、2015 年引入 NLP）的破局方式**：先扫语料找高频相邻字节对不断合并，形成常用子词单元；对于生造新词则自动退化拆回字节级零件——**永远不会出现 UNK**。训练只需要 CPU，成本极低，这正是 BPE 的灵魂所在。

**Q：词表和模型是什么关系？**
完全绑定。词表决定了 Embedding 层的行数（`vocab_size × hidden_dim`）和最后 LM Head 的输出维度。换词表就等于换模型，两者不能兼容。这也是为什么开源社区同款权重必须配套同款 tokenizer 使用。

**Q：为什么用 BPE + UTF-8 字节级的组合？**
UTF-8 字节序列可以无损表示任何语言的任意字符（中文、日文、Emoji 全覆盖）。以字节为原子底层，再让 BPE 在上层学习高频合并，最终得到一个**跨语言、无盲区**的词表，让模型自己去学习词表之间的规律。

---

## 2. 文字如何变数字、又如何变回文字？

**Q：模型的输入输出到底是什么？**
整条流水线：`文字 → Tokenizer 切分 → Token ID 数字序列 → 模型（纯数字矩阵运算）→ 输出 Token ID 数字 → Decoder 解码 → 文字`。模型内部从头到尾只认识数字，不知道也不关心汉字长什么样，只知道每个 Token 的编号是多少。

---

## 3. SFT：少量高质量数据为何能改变模型性格？

**Q：为什么 SFT 数据量不需要很大？**
2023 年 Meta 的 LLaMA 系列论文证明：预训练阶段已经让模型拥有了大量世界知识，SFT 的作用不是”灌知识”，而是**激活并对齐输出格式与风格**——从”接龙学究”变成”听话助手”。因此几万条精雕细琢的高质量问答，效果往往远超几十万条粗糙数据。

**Q：SFT 有系统提示词（System Prompt），训练时为什么要对它们做 `-100` 掩码？**
系统提示词是人类写好的”角色设定”，不是模型需要学会生成的内容。如果让损失函数对提示词部分也扣分，模型会浪费算力去死记背诵人类的任务描述，反而忘记学习”如何生成高质量回答”这件真正重要的事。`-100` 掩码让 `CrossEntropyLoss` 在提示词和用户发问区域直接失明，只对 Assistant 吐出的答案追责。

**Q：SFT 全参数微调（Full Fine-Tuning）和 PEFT 有什么区别？**

- **Full SFT**：所有参数全部更新，效果最好，但成本极高（显存需求接近预训练量级）；
- **PEFT（参数高效微调）**：以 LoRA / QLoRA 为代表，只训练极少数新增的低秩矩阵参数，原始权重完全冻结。用 1%~5% 的参数量撬动约 90% 的微调效果，是个人和小团队的首选方案。

---

## 4. RLHF：SFT 之后为什么还要”人类反馈”？

**Q：SFT 不够好吗？**
SFT 有两个根本瓶颈：

1. **标注成本**：让标注员写一条高质量的标准答案耗时耗力；但让他在两个答案里选”哪个更好”却极其轻松——偏好排序的成本远低于生成答案；
2. **覆盖盲区**：SFT 只教模型”该说什么”，对于似是而非的有害或误导性回复，覆盖面永远不够。

**RLHF 三步走战略：**


| 阶段                           | 做什么                                           | 核心价值                               |
| :----------------------------- | :----------------------------------------------- | :------------------------------------- |
| **第一步：SFT 热身**           | 用指令数据把 Base 模型调成能对话的初代助手       | 给后续强化学习提供基础起点             |
| **第二步：训练奖励模型（RM）** | 收集人类偏好排序数据，训练一个”打分裁判”小模型 | 把人类主观口味量化成可优化的分数信号   |
| **第三步：PPO 强化学习**     | 让 SFT 模型和冻结的 RM 博弈，最大化奖励分值      | 在不把模型练废的前提下持续逼近人类偏好 |

> **DPO 是 RLHF 的轻量替代方案**：跳过训练奖励模型这一步，直接用偏好数据对模型进行对比优化，更简洁、更稳定，是当前业界最热门的对齐技术之一。

---

## 5. DDP 多卡训练的本质

**Q：DDP 分数据、多卡、最后梯度平均，具体怎么做到的？**
每张显卡各自持有模型的完整副本，但负责处理不同的数据分片。每次反向传播结束后，所有卡通过高速互联（NVLink/InfiniBand）对梯度做全局平均（All-Reduce），然后各自用同一份平均梯度同步更新参数。**数据吞吐量线性扩展，模型行为完全等价于单卡大 Batch 训练。**

---

## 6. LoRA vs QLoRA：省显存的两条路怎么选？

**Q：LoRA 和 QLoRA 到底有什么区别？**

核心区别只有一个：**基础模型的权重用什么精度存着**。


| 对比维度             | LoRA                            | QLoRA                                      |
| :------------------- | :------------------------------ | :----------------------------------------- |
| **基础模型精度**     | FP16 / BF16（半精度，正常加载） | NF4（4-bit 正态浮点，量化后加载）          |
| **新增 LoRA 分支**   | BF16 精度，正常训练             | BF16 精度，正常训练                        |
| **原始权重是否冻结** | 是，不参与梯度                  | 是，不参与梯度                             |
| **显存占用**         | 比全参数少 60%~70%              | 比 LoRA 再少约 60%，可在 24G 上跑 65B 模型 |
| **训练速度**         | 较快                            | 因量化/反量化多一步，约慢 30%              |
| **精度损失**         | 几乎无                          | 极小，大多数任务可接受                     |
| **典型使用场景**     | 单卡 24G 跑 7B / 13B 微调       | 单卡 24G 跑 33B / 65B 微调                 |

**Q：QLoRA 为什么能省这么多显存？**
省显存的逻辑分三层叠加：

1. **基础权重量化**：把冻结的基础模型从 FP16（每参数 2 字节）压到 NF4（每参数 0.5 字节），直接省 75% 的权重显存；
2. **只训练 LoRA 参数**：优化器状态（Adam 的 $m$ 和 $v$）只为 LoRA 的极少数参数维护，不为基础模型的几十亿参数维护；
3. **分页优化器（Paged Optimizer）**：QLoRA 论文还引入了 CPU 内存换页技术，在 GPU 显存不足时把优化器状态临时换到 CPU 内存，防止 OOM。

**Q：实际工程中怎么选？**

- 显卡显存够（比如 24G 跑 7B）→ 优先 **LoRA**，更快更简单；
- 想用小显卡跑大模型（比如 24G 跑 13B+）→ 上 **QLoRA**，接受一点速度换空间的代价；
- 有多卡资源 → 直接上 **Full SFT + DDP**，效果上限最高。

**权重矩阵 W 不需要重新计算**——它是模型参数，推理时是固定的。

**需要计算的是矩阵乘法 `x @ W`**：

```
无KV Cache时，每生成一个新token：
Step 1: x = [token₁]           → K₁ = x₁ @ W_k,  V₁ = x₁ @ W_v
Step 2: x = [token₁, token₂]   → K₁ = x₁ @ W_k,  K₂ = x₂ @ W_k  ← x₁ @ W_k 重复算了！
Step 3: x = [token₁, token₂, token₃] → K₁, K₂, K₃ 全部重新算  ← K₁, K₂ 又重复算了！
```

**"重复计算"指的是**：虽然 `x₁`（第一个token的embedding）和 `W_k` 都没变，但每一步都要重新执行 `x₁ @ W_k` 这个矩阵乘法。

## 有KV Cache后

```
Step 1: K₁ = x₁ @ W_k → 缓存 K₁
Step 2: K₂ = x₂ @ W_k → 拼接 [K₁, K₂]（K₁直接从缓存取，不用再算 x₁ @ W_k）
Step 3: K₃ = x₃ @ W_k → 拼接 [K₁, K₂, K₃]
```
