## 🚀 Day 6：功能突破与深度集成

### 1. 核心功能实现

- **流式输出 (Streaming)**：引入基于 `astream_events` 的检测模型，实现响应内容的实时流式推送。
- **用户画像保存 (Tool Calling)**：通过工具调用实现对用户偏好和事实的自动识别与长期保存。

### 2. 技术深挖：大模型工具调用原理

LangChain 会将工具定义转化为严格的 **JSON Schema** 格式，并将其作为 System Prompt 的一部分强制输入。

#### 模型如何理解并执行工具？

1. **结构化能力**：现代模型（如 Qwen, GPT-4等）在预训练和微调阶段经过了海量的“指令 -> JSON”语料训练。一旦察觉到任务需求，模型会立刻停止“说人话”，转而输出一段严格的机器代码（JSON格式）。
2. **底层翻译**：当你使用 `@tool` 定义 Python 函数时，LangChain 底层会把它翻译为符合国际标准的 JSON Schema 发给模型。
3. **闭环流程**：
   - 模型返回 `tool_calls` 数据包（包含具体的函数名和参数）。
   - 框架（如 LangGraph）拦截并执行本地 Python 函数。
   - 执行结果返回给模型，由其总结后最终答复用户。
4. **自我纠错机制 (Self-Correction)**：
   - 如果模型生成的 JSON 报错（如缺失必填参数），LangGraph 会捕捉报错信息（如 `Error: missing required argument 'value'`）直接扔回给模型。
   - 模型看到报错后会自动意识到错误，并重新发送一份正确的 JSON 请求。

#### 行业里程碑

- **逻辑框架**：2022年底的 ReAct 论文（Reasoning + Acting）。
- **理论起源**：Meta AI 2023年2月论文《Toolformer》。
- **工业落地**：2023年6月 OpenAI 正式发布 Function Calling。

---

### 2. 面试级深度解析

---

#### ❓ Q1：为什么不直接让 LLM 一步完成，要搞多智能体？

**A：** 单个 Agent 面对复杂任务有两大瓶颈：

1. **上下文窗口限制**：一次解决"数据分析 + 写报告 + 生成代码"，所有内容挤在一个 Prompt 里，质量严重下滑。
2. **角色混乱**：同一个模型既要分析数据又要写文章，提示词互相干扰，专注度不足。

多智能体的核心思想是**分工专业化**：每个 Agent 只做一件事，配备专属 System Prompt 和工具集，就像公司里的部门协作——财务、技术、运营各司其职。

---

#### ❓ Q2：Supervisor 的 Plan 过程具体怎么工作的？

**A：** Supervisor 接收用户输入后，调用 LLM 输出结构化 JSON：

```json
{
  "analysis": "用户需要先查数据，再写报告",
  "tasks": [
    {"id": 1, "description": "查询销售数据并分析趋势", "assigned_to": "data"},
    {"id": 2, "description": "根据分析结果撰写报告", "assigned_to": "writer"}
  ]
}
```

关键设计：
- **强制 JSON 输出**：System Prompt 明确要求格式，配合解析容错（try/except），失败则降级用 `data` Agent 兜底。
- **有序列表**：`tasks` 数组按执行顺序排列，后一个 Worker 的 `handoff_context` 包含前一个 Worker 的输出。
- **单次规划**：Plan 只在首次调用时生成，后续 Supervisor 只负责"找下一个 pending 任务"，不重复规划。

---

#### ❓ Q3：一次 Plan 之后，Supervisor 是怎么"依次"调度 Worker 的？

**A：** 这里的关键是 **LangGraph 的循环图结构**：

```
START → supervisor → route_to_worker
                       ├─ code_agent  ─┐
                       ├─ data_agent  ─┼─→ increment → supervisor (回环！)
                       ├─ writer_agent─┘
                       └─ aggregator → END
```

每次 Supervisor 被调用：
1. 扫描 `task_plan`，找到第一个 `status == "pending"` 的任务
2. 把它改为 `"in_progress"` 并设置 `current_worker`
3. 路由函数 `route_to_worker` 根据 `current_worker` 分发给对应 Agent
4. Worker 执行完毕后走 `increment → supervisor`，进入下一轮循环
5. 所有任务 `completed/failed` 后，返回 `"FINISH"` 路由到 Aggregator

本质：**Supervisor 是一个有状态的迭代器**，每次只推进一步，靠图的回环驱动整个流程。

---

#### ❓ Q4：为什么要加 Re-plan？直接跳过失败的任务不行吗？

**A：** 直接跳过的问题在于**任务之间有依赖关系**。

举例：
```
Task 1: [data] 从知识库查询 2024 年销售数据     → 失败（索引为空）
Task 2: [writer] 根据销售数据写分析报告          → 前置数据不存在，一定废
Task 3: [code] 生成数据可视化图表代码            → 同上
```

如果直接跳过 Task 1，Task 2、3 会生成"空气报告"——输出了但没有价值。

Re-plan 的价值在于 **LLM 能理解失败原因并调整策略**：
- 可能改为：先用搜索工具获取数据，再分析再写报告
- 可能降级：直接告知用户"数据不可用，以下报告基于通用知识"
- 可能拆分：把大任务拆成更小的可执行步骤

这是多智能体系统比单纯流水线更智能的核心体现——**遇到问题会思考，不是死板执行**。

---

#### ❓ Q5：Re-plan 的具体实现是什么？代码层面怎么触发的？

**A：** 触发链路：

```python
# base.py — Worker 执行后检测
is_success, failure_reason = self._check_task_success(result, error)
if not is_success:
    needs_replan = True
    replan_reason = f"[{self.name}] {failure_reason}"

return {
    "needs_replan": needs_replan,   # 写入共享状态
    "replan_reason": replan_reason,
    "task_plan": updated_plan,      # 任务状态标记为 "failed"
}
```

```python
# supervisor.py — 下一次被调用时检测
if needs_replan and task_plan:
    if replan_count >= MAX_REPLAN_COUNT:
        # 超限，强制继续
        return {"needs_replan": False, ...}

    # 构建重规划 Prompt：包含已完成任务 + 失败任务 + 失败原因
    response = llm.invoke(replan_messages)
    # 解析新计划，保留已完成任务，追加新任务
    new_task_plan = [t for t in task_plan if t.get("status") == "completed"]
    # 将新任务追加并继续执行
```

状态流转：`needs_replan=True` → supervisor 拦截 → LLM 重规划 → 新 task_plan → 继续图循环。

---

#### ❓ Q6：Re-plan 会不会无限循环？怎么保护？

**A：** 双重保护机制：

| 保护层 | 变量 | 默认值 | 作用 |
|--------|------|--------|------|
| Re-plan 次数上限 | `MAX_REPLAN_COUNT` | 2 | 最多重规划 2 次，超出强制继续 |
| 总迭代轮次上限 | `MAX_ITERATIONS` | 10 | 整个图最多循环 10 次，超出直接跳 Aggregator |

两层都在 `graph.py` 和 `supervisor.py` 中分别实现，即使 Re-plan 本身也失败了，最多再循环几次后由迭代计数器强制终止，不会死循环。

---

#### ❓ Q7：Worker 之间怎么传递上下文？

**A：** 通过 State 中的 `handoff_context` 字段：

```python
# base.py — Worker 完成后
return {
    "handoff_context": result,   # 把自己的输出作为下游的输入上下文
    "worker_results": worker_results,
}
```

下游 Worker 的 System Prompt 会自动拼接：

```python
if handoff_context:
    full_prompt += f"\n\n## 上游 Agent 传递的上下文:\n{handoff_context}"
```

这样 Writer Agent 在写报告时，能看到 Data Agent 分析出的数据；Code Agent 能看到 Writer 描述的图表需求。

## 🚀 Day 10：全局幻觉修复与泛化提示词重构

### 1. 核心幻觉问题定位

- **现象**：当询问实时性极强的数据（如“今天的A股开盘点数”）时，模型可能产生严重幻觉。
  - **身份幻觉**：强行自称是 DeepSeek，并坚持其自身知识库截止于 2024 年。
  - **时间认知冲突**：即便系统存在获取时间的工具，但模型因自身世界观尚未跨越 2024 年，遇到 2026 年的时间会产生不信任导致的“拒绝回答”，甚至在多智能体模式下因查询不到2024数据而引发无限 Re-plan，导致 Token 飞速消耗殆尽。

### 2. 重构思路（从“强制”走向“泛化心智约束”）

此前的尝试是：硬编码大模型名字（必须叫通义千问）+ 暴力把当天的具体字符串插入所有上下文。但这样通用性极差，模型换了或者场景换了就崩溃。

现在的方案转向**全局身份和规则重构**（作用于 `main.py` 以及多智能体的 `supervisor.py` 和 `data_agent.py`）：
1. **剥离基座身份执念**：加入规则要求“这只是一个客观解决问题的工具箱大脑，不要主动辩论底层模型身份（无论是Qwen还是DeepSeek），以及永远不要主动提知识库截止时间”。
2. **强制移交信任**：强行给 LLM 注入认知设定：“由于使用了现代智能体框架，你的知识不受静态训练数据限制，你**必须无条件信任外部工具（特别是时间和搜索 API）吐出的现实物理数据**”。
3. **“时间锚点”自动前置法则**：
   - 只要任务涉及“今天”、“实时”或“数据查询”，不管用户给没给时间，**第一步必定先调用 `get_current_datetime` 拿到真实的物理时间**。
   - 随后再调用 `web_search` 工具时，强制把刚拿到的今天日期**硬拼到搜索关键词中**（如 "2026年3月17日 A股点数"）。这使得搜索引擎能精准返回当天的最新爬虫数据，而不是泛泛的过时页面。

### 3. Debug 效果验证

经过重构系统提示词架构后，无论是单智能体还是多智能体（Data/Supervisor）均能在遇到时效性请求时：
- 不再自称 ChatGPT/DeepSeek；
- 主动查询 2026 年的时间并以此为第一依据；
- 正确组装搜索关键词并返回全网真实的 A 股开盘报表数据。
